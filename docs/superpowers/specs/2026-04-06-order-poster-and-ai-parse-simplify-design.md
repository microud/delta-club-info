# 订单海报 & AI 解析简化 设计文档

## 背景

俱乐部推广图片格式多样，AI 解析复杂海报（尤其护航趣味单）成功率低。为降低复杂度：
1. AI 解析聚焦结构明确的服务类型，护航趣味单允许解析失败
2. 新增"订单海报"功能，俱乐部可维护多张原始海报图片存储在 OSS，供后续小程序展示

## 改动范围

### 1. AI 解析调整

**当前状态**：AI 尝试从图片中解析所有服务类型，护航趣味单因玩法多样、规则复杂，解析效果不稳定。

**改动**：
- 调整 AI prompt，明确标注护航趣味单为"尽力解析"，允许该部分返回空
- 跑刀、陪玩、护航体验、护航标准继续作为主要解析目标
- 不改变现有的数据结构和解析流程

**改动文件**：
- `apps/server/src/ai-parse/ai-parse.service.ts` — 调整 prompt 内容

### 2. 订单海报功能

#### 数据模型

Club 表新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| orderPosters | text[] | 订单海报图片 URL 数组，默认空数组 |

使用 PostgreSQL 原生数组类型，每个元素为 S3 存储的图片 URL。

#### 后端

- Club schema 新增 `orderPosters` 字段（`text('order_posters').array()`，默认 `[]`）
- Club CRUD 接口自然支持该字段的读写，无需新增接口
- 图片上传复用现有 `/admin/upload` 通用上传接口
- 俱乐部详情 API 返回该字段（为小程序预留）

#### Admin 前端

- 俱乐部编辑表单（`club-form.tsx`）新增"订单海报"区域
- 支持多张图片上传、预览、删除、排序
- 上传调用现有 `/admin/upload` 接口获取 URL，保存时将 URL 数组写入 `orderPosters` 字段
- 俱乐部列表页不展示该字段

#### 存储

- 复用现有 StorageModule 和 S3 配置
- 图片存储路径：`clubs/{clubId}/posters/{filename}`

## 不涉及的改动

- 小程序端展示（后续单独处理）
- 企业微信消息接收流程
- 服务定价的数据结构
- 护航趣味单的手动录入流程（已有）
